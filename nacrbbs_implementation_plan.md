# NACR BBS 项目优化与 API Key 功能实施说明

## 1. 项目概述

本项目是一个 Nuxt 3 双端应用，主要分为：

- `web/client`：用户端
- `web/admin`：管理端
- `app`：后端 Fastify 服务与业务控制器

当前前后端通过统一的 API 访问层进行通信，用户端使用 `UToken` cookie 透传用户身份，管理端使用 `AToken` cookie 透传管理身份。后端已有基本的用户鉴权、管理鉴权、帖子、评论、上传、消息等业务模块。

本次工作目标分为三部分：

1. 前端性能优化与部署优化
2. 用户端 API Key 管理功能
3. API Key 鉴权的独立 REST 接口

同时需要对最近提交的代码进行审计，发现问题直接修复，并保持每个逻辑变更独立提交。

## 2. 当前已确认的技术架构

### 2.1 前端

- Nuxt 3 双端：`web/client` 与 `web/admin`
- UI 组件体系：Arco Design Vue
- 状态管理：Pinia
- API 访问封装：`useApiFetch()`
- SSR 请求封装：`useApiSsr()`
- 登录态透传方式：cookie 中的 `UToken` / `AToken`

### 2.2 后端

- Node.js + Fastify
- MySQL2 作为主数据层
- Redis 用于缓存、限流、临时状态
- 后端鉴权机制基于 AES-256-CBC 加密 token
- 用户、管理员、公开接口通过不同中间件组区分

### 2.3 现有鉴权模式

- `NoMestUTokenVar`：公开接口
- `UTokenVar`：用户 token 鉴权接口
- `ATokenVar`：管理员 token 鉴权接口

当前需要新增：

- `ApiKeyVar`：API Key 鉴权接口

## 3. 已发现的问题与优化方向

### 3.1 前端性能问题

#### 3.1.1 组件级响应式开销

`web/client/components/CommentItem.vue` 中存在对 `props.data` 的深度 watch，并且在 watch 内部直接回写 `props.data`。这会带来：

- Vue 警告风险
- 不必要的深度响应式追踪
- 额外的渲染开销

这个 watch 应删除，直接依赖 props 的响应式即可。

#### 3.1.2 重组件未充分懒加载

`web/admin/components/MonacoEditor.vue` 是典型重组件，虽然内部做了动态 import，但如果页面级仍直接同步挂载，会影响首屏和页面切换速度。

建议：

- 页面中按需使用 `LazyMonacoEditor`
- 仅在真正进入编辑场景时再加载
- 避免多个页面重复实例化 Monaco

#### 3.1.3 API Base URL 硬编码

`web/client/app-config.js` 与 `web/admin/app-config.js` 中硬编码了后端地址，这不利于部署与环境切换。

建议迁移到 Nuxt `runtimeConfig.public.apiBase`：

- 开发环境、测试环境、生产环境可独立配置
- 构建产物无需改源码即可切换目标服务
- 更适合服务器部署与多环境发布

#### 3.1.4 重复请求与缓存不足

当前部分页面会重复获取：

- 用户信息
- 站点配置
- 统计信息
- 详情页数据

建议：

- 对公共且短期稳定数据增加短缓存
- SSR 场景复用已有数据
- 站点配置、myInfo 等采用统一缓存策略
- 详情页结果结合 key 进行短期缓存

#### 3.1.5 请求封装可进一步统一

当前 `useApiFetch()` 中仍存在手动 `JSON.parse(res.data.value)` 的情况，导致：

- 错误处理分散
- 解析逻辑重复
- 不利于统一控制缓存、超时和错误提示

建议后续收敛为统一请求封装，并统一处理：

- JSON 解析
- HTTP 状态判断
- 401/403 跳转
- 重试或错误消息

### 3.2 部署与包体优化

#### 3.2.1 前端压缩

用于服务器部署时，建议确认构建产物压缩策略：

- Nuxt 生产构建压缩开启
- JS/CSS 压缩
- 静态资源压缩或 CDN 处理
- 大页面按路由拆包

#### 3.2.2 无关目录清理

部署前应避免把以下内容带入生产发布包：

- `node_modules`
- `.nuxt`
- `.output`
- 临时构建缓存

这可以减少部署体积，也能降低服务器磁盘和内存压力。

#### 3.2.3 静态资源与图片优化

建议检查：

- 图片是否有过大尺寸
- 头像、封面、附件是否需要压缩
- 是否能使用 WebP/AVIF
- 静态资源是否可以走浏览器缓存或 CDN

### 3.3 内存与运行时优化

#### 3.3.1 组件懒加载

对以下类型组件优先做条件加载：

- Monaco 编辑器
- 富文本编辑器
- 图表组件
- 大型弹窗
- 低频使用的管理端模块

#### 3.3.2 列表页字段控制

管理端表格页、帖子列表、评论列表等接口应避免一次返回过多字段，建议：

- 列表只返回必要字段
- 详情页再单独拉取扩展字段
- 大字段内容延迟加载

#### 3.3.3 只读缓存对象

对不会频繁变化的内容可以使用短时缓存：

- 站点配置
- 分类列表
- 热门统计
- 只读详情数据

## 5. API Key 功能设计

### 5.1 功能目标

用户需要一个类似 GitHub 的 API Key 管理能力，支持：

- 创建 API Key
- 查看 Key 列表
- 删除 / 失效 Key
- 设置 Key 名称
- 设置 Key 权限范围
- 设置 Key 有效期
- 创建后只展示一次明文 Key

同时需要提供一个只通过 API Key 鉴权的 REST API，用于快速执行用户自身允许的操作，例如：

- 发帖
- 评论
- 点赞
- 读取自己的数据
- 快捷自动化操作

### 5.2 设计原则

- API Key 只代表用户的授权子集，不等同于完整登录态
- Key 必须绑定到具体用户
- Key 必须有过期时间或永久策略
- Key 必须有可控的权限范围
- Key 必须支持失效与删除
- 所有 API Key 调用都应可审计

### 5.3 权限范围建议

可先按功能域设计 scope，例如：

- `threads:read`
- `threads:write`
- `threads:delete`
- `comments:read`
- `comments:write`
- `likes:write`
- `profile:read`
- `profile:write`

也可以按更粗粒度划分：

- `read`
- `write`
- `post`
- `comment`
- `like`
- `manage_own`

建议在数据库中存储为可解析的字符串或 JSON 数组，便于后续扩展。

### 5.4 有效期设计

建议支持：

- 永久
- 7 天
- 30 天
- 90 天
- 自定义日期

过期后自动失效，不允许继续调用接口。

### 5.5 安全要求

- Key 只在创建时返回一次明文
- 数据库存储应避免直接明文保存
- 每个 Key 必须可单独撤销
- Key 调用应做限流
- 建议记录调用 IP、UA、路径、时间
- 关键写操作建议记录审计日志

## 6. 后端实现方案

### 6.1 API Key 数据层

当前项目已有 `n_apikeys` 表雏形，建议补充并规范以下字段：

- `id`
- `n_key`：密钥值或其哈希
- `n_uid`：所属用户 ID
- `n_time`：创建时间
- `n_type`：类型或状态
- `n_white`：白名单 / scope 信息
- `n_black`：黑名单 / 禁用信息
- `n_name`：Key 名称
- `n_ipstatus`：IP 限制或状态

如需更完整设计，可新增：

- `n_expire_at`：过期时间
- `n_last_used_at`：最后使用时间
- `n_status`：启用 / 禁用
- `n_scopes`：权限范围
- `n_note`：备注

### 6.2 用户端 API Key 控制器

`app/controller/user/Apikey.js` 需要增强为：

- 仅允许用户查看自己的 Key
- 创建时生成安全随机 Key
- 保存权限范围与有效期
- 列表只返回脱敏信息
- 删除时校验归属
- 编辑时校验归属与字段合法性

### 6.3 ApiKeyVar 中间件

新增 `app/router/Ware/ApiKeyVar.js`，其职责：

- 读取请求头中的 API Key
- 查表验证 Key 是否存在
- 校验是否属于启用状态
- 校验是否过期
- 校验请求路径是否匹配 scope
- 注入 `request.Ware`，供后续控制器读取用户身份和授权信息

### 6.4 独立 REST API

需要在路由中新增一组只接受 API Key 的接口，例如：

- `POST /api/threads/add`
- `POST /api/threads/comment`
- `POST /api/threads/like`
- `GET /api/threads/detail`

具体暴露哪些接口，应按“用户自身权限可执行操作”来限定，避免给出过强权限。

### 6.5 旧接口复用

建议优先复用现有控制器逻辑，例如：

- `threads.js`
- `comment` 相关方法
- `like` 相关方法

只是在路由层和鉴权层做区分，避免重复实现业务逻辑。

### 7.1 用户端入口

用户中心已有“我的动态 / 我的资料”等入口，适合新增：

建议新增页面：

页面功能包括：

- API Key 列表展示
- 创建 Key 表单
- 名称输入
- 权限范围多选
- 有效期选择
- 删除 / 停用操作
- 创建后展示一次明文 Key 并提示保存

### 7.3 页面交互原则

- 创建成功后立即展示明文 Key
- 关闭弹窗后不再次展示明文
- Key 列表默认隐藏完整值，仅显示后缀或掩码
- 删除操作应有确认提示
- 表单字段应显示可选范围和说明

## 9. 验收标准

### 9.1 性能优化验收

- 首页首屏请求减少
- 用户中心重复请求减少
- 重组件按需加载生效
- 构建产物可用于服务器部署
- 不再把 `node_modules`、`.nuxt`、`.output` 带入提交或发布包

### 9.2 API Key 功能验收

- 用户可创建 API Key
- Key 可设置名称、权限和过期时间
- Key 列表仅能查看自己的数据
- 删除后立即失效
- 创建后只显示一次明文 Key
- API Key 鉴权接口可独立调用
- API Key 不能越权访问其他用户数据

### 9.3 审计验收

- 最近提交无明显回退
- 登录、发帖、评论、上传、管理操作正常
- 权限校验与过期校验生效
- 前端无明显 Vue 警告与重复渲染问题

---

## 10. 风险点

- API Key 的权限模型设计过粗会导致后续难扩展
- 只存明文 Key 会有安全风险，应尽量存哈希或加密值
- 独立 REST API 如果复用了过强的控制器能力，可能放大权限面
- 前端缓存过度可能导致数据陈旧，需要设置合理 TTL
- 重组件懒加载若处理不当，可能影响交互首次响应

---

## 11. 结论

当前项目具备继续扩展 API Key 能力的基础，但仍需要补齐：

- 前端性能优化
- 配置统一
- API Key 数据层与鉴权层
- 用户端管理界面
- 独立 REST API

建议先完成基础功能，再进行性能优化和回归审计，最后按逻辑拆分提交并推送到远端仓库。
